1Core QE Principles & Agile Ceremonies

The QE Framework embeds quality at every stage of delivery — from the moment a story is written through to production monitoring. Quality is a team responsibility, not a QA afterthought.

💡
Core PurposeTurn quality from a QA afterthought into a team-owned practice embedded at every stage of delivery — shift-left, automate-first, and measure everything.

Core Principles

PrincipleWhat It Means in Practice
Shift-LeftQE joins at story creation — not after dev is done
Quality as a Team SportBA, Dev, QA, and DevOps all own quality outcomes
Risk-Based ThinkingTest effort proportional to business impact and technical risk
Automation-FirstAutomate stable regression; manual effort for exploration and new features
Continuous FeedbackQuality signals visible at every stage: PR, merge, nightly, release
Living DocumentationBDD scenarios and test results always current and linked to Jira
Fail FastDetect defects at the earliest possible pipeline stage
Testability by DesignDev and QA jointly ensure features are observable, controllable, and isolatable

2Three Amigos Sessions (BA / Dev / QA)

The Three Amigos aligns all three perspectives on a story before it enters a sprint — when misunderstandings are cheapest to fix. The goal is shared understanding, written in testable scenarios.

Who Attends and Why

RolePerspectiveKey Contribution
BA / POWhat and WhyBusiness intent, user value, acceptance criteria, compliance needs, out-of-scope definition
DeveloperHowTechnical approach, constraints, failure modes, integration points, schema changes, async risks
QAWhat could go wrongNegative paths, edge cases, data risks, regression exposure, testability of AC
ℹ️
Optionally invite: UX Designer (UI stories), DevOps (infrastructure changes), Security (sensitive data stories).

Session Agenda — 60 Minutes

1
0–5 min: BA presentsStory goal, user value, and acceptance criteria as written
2
5–15 min: Dev raisesTechnical approach, risks, failure modes, integration points
3
15–30 min: QA raisesNegative paths, edge cases, data requirements, environment needs
4
30–45 min: GroupWrite Gherkin scenarios together — happy path, negative, edge cases
5
45–55 min: AgreeFinal AC, create sub-tasks, log unknowns and risks
6
55–60 min: ConfirmDoR — READY to enter sprint, or BLOCKED with specific reason

Perspective Cheat Sheet — Questions Each Role Asks

🗂 BA Asks

  • Are all AC lines testable and unambiguous? (no "valid", "correct", "fast")
  • What is the measurable business outcome?
  • Compliance, audit, or accessibility requirements?
  • What is explicitly out of scope?
  • Any upstream/downstream dependency impacts?

⚙️ Dev Asks

  • What existing services or code are touched?
  • DB migrations, schema changes, async processes?
  • Concurrency, caching, or race condition risks?
  • What unit-testable logic is introduced or changed?
  • Performance implications at scale?

🧪 QA Asks

  • What data is needed — does it exist in QA environments?
  • Boundary values, null/empty inputs, max lengths?
  • What existing features could break?
  • Can every AC line become a verifiable test?
  • Rollback plan if this breaks in production?

✅ Output — Update Jira Story

  • AC refined using Given/When/Then
  • Negative paths and edge cases added
  • NFRs noted (performance, security, a11y)
  • Test data requirements documented
  • Sub-tasks: Test, Automation, Environment Setup
  • Risks/unknowns logged as comments
  • DoR confirmed or story blocked with reason

3Definition of Ready & Done

Definition of Ready (DoR)

ALL must pass before a story enters a sprint.

  • User story format: "As a [role], I want [goal], so that [value]"
  • Minimum 2 AC lines — each specific and testable
  • No vague language: valid, appropriate, fast, easy
  • UI/API designs or contracts available (if applicable)
  • Dependencies on other teams or services identified
  • Test data requirements documented
  • NFRs noted or deferred with PO sign-off
  • Three Amigos completed or waived with PO sign-off
  • Story is sprint-sized — buildable and testable in one sprint

Definition of Done (DoD)

ALL must pass before a story is marked Done.

  • All AC pass — verified by QA
  • Unit tests written and passing (≥ 80% coverage on changed code)
  • Functional tests cover all AC (happy + negative paths)
  • E2E tests updated for impacted critical journeys
  • API/contract tests updated if interfaces changed
  • Exploratory testing completed — notes logged in Jira
  • No open Critical or High defects linked to story
  • Test results attached or linked in Jira
  • Accessibility spot-checked (WCAG 2.1 AA)
  • Performance baseline verified if performance-critical
  • Security review completed if story touches auth, data, or APIs

5API & Contract Testing

APIs are the primary integration surface in modern systems. QE co-owns the API test layer alongside Dev. Testing here catches integration defects faster and cheaper than E2E tests.

API Test Design Patterns

PatternPurposeExample
Happy pathValid request → expected responsePOST /orders → 201, order ID returned
Schema validationResponse matches OpenAPI contractAll required fields, correct data types
Negative / errorInvalid input → correct error codeMissing auth → 401 Unauthorized
BoundaryEdge values for fieldsMax payload, empty array, null
Auth & permissionsRole-based endpoint accessViewer token → 403 on write endpoint
IdempotencyRepeat request is safePUT twice → no duplicate side effect
Stateful sequenceMulti-step API flowCreate → Fetch → Update → Delete
Timeout & retrySlow/unavailable dependency504 Gateway Timeout handled gracefully

Consumer-Driven Contract Testing (Pact)

Pact prevents integration breakage between services without requiring full E2E runs. The consumer defines expectations; the provider verifies them on every build.

1
Consumer team writes Pact testsDefine expected API interactions — what requests they send and what responses they expect
2
Pact Broker stores the contractContract file published and versioned centrally (Pact Broker or Pactflow)
3
Provider CI verifies on every buildThe API service runs against the consumer's contract — breaking change = CI failure
4
Integration breakage caught earlyBefore any E2E test runs — fast feedback loop, no environment dependency

6BDD Gherkin & ATDD Test Case Generation

🔄
The ATDD Shift
Old: Story → Dev builds → QA tests → Defects found late.
ATDD: Three Amigos write scenarios → Dev implements to make them pass → QA verifies → Scenarios become automation.

Who Uses Gherkin and How

RoleResponsibility
BA / POReviews and signs off scenarios as official AC before sprint — the scenarios ARE the acceptance criteria
DeveloperUses Given state to set up unit test preconditions; implements code to make the Then clause pass
QAOwns scenario completeness; executes manually first; tags @automation-candidate when stable
Automation EngineerWrites step definitions for all @automation-candidate scenarios after manual sign-off

Gherkin Writing Rules

✅ DO

  • Write in business language — no technical jargon
  • One behaviour per scenario
  • Use present tense: "I should see" not "I will see"
  • Make Given state fully explicit — no assumed setup
  • Name scenarios: [behaviour] when [condition]

❌ DO NOT

  • Describe UI mechanics: "click the blue button"
  • Use vague words: valid, correct, appropriate
  • Chain unrelated actions in one scenario
  • Duplicate scenarios across feature files
  • Use And more than twice — split the scenario

AC to Scenario Mapping — Minimum Required

AC TypeMinimum Scenarios
Happy path1 success + 1 data variation
Validation rule1 passing + 1 failing per rule
Permission / role1 per role (authorised + unauthorised)
Error / exception1 per distinct error condition
Integration point1 success + 1 dependency-down failure
NFR (perf, accessibility)1 per requirement tagged @manual or @performance

Tagging Strategy

TagPurposeRun In
@smokeMust-not-break critical pathsEvery deploy
@regressionFull scenario suiteNightly
@happy-pathAC success scenariosQE planning
@negativeError/rejection scenariosQE planning
@edge-caseBoundary conditionsQE planning

Worked Example — Password Reset Feature File

Feature: Password Reset via Email
  As a registered customer
  I want to reset my password via email
  So that I can regain access if I forget my credentials

  Background:
    Given the user "alex@example.com" is registered
    And no active reset token exists for "alex@example.com"

  @happy-path @smoke @automation-candidate
  Scenario: Registered user receives a password reset link
    Given I am on the Forgot Password page
    When  I enter email "alex@example.com" and submit
    Then  I should see "Check your inbox for reset instructions"
    And   a reset email is sent to "alex@example.com"
    And   the link expires in 24 hours

  @negative @automation-candidate
  Scenario: Unregistered email shows no-account message
    When  I enter "unknown@example.com" on the Forgot Password page
    Then  I should see "No account found with that email address"
    And   no reset email is sent

  @edge-case @automation-candidate
  Scenario Outline: Password complexity rules enforced
    Given a valid token exists for "alex@example.com"
    When  I enter password "<password>"
    Then  I should see "<error>"

    Examples:
      | password  | error                                        |
      | pass      | Must be at least 8 characters               |
      | password1 | Must contain at least one uppercase letter  |
      | PASSWORD1 | Must contain at least one lowercase letter  |
      | Password  | Must contain at least one number            |

  @edge-case @security @automation-candidate
  Scenario: Rate limit blocks excessive reset requests
    Given "alex@example.com" has requested 3 resets this hour
    When  I request another reset
    Then  I should see "Too many attempts -- try again in 60 minutes"
    And   no email is sent

9CI/CD Quality Gates & Pipeline

Every code change flows through five progressive quality gates. A gate failure stops the pipeline — the team fixes forward, never backward.

1
Gate 1 — PR Gate (pre-merge) Unit tests + SAST scan | Code coverage ≥ 80% on changed code | Zero new critical security findings
FAIL: PR blocked — dev fixes before review can proceed
2
Gate 2 — Merge Gate (post-merge) Functional tests + API tests | Pact contract verification | DAST baseline scan (OWASP ZAP)
FAIL: Branch blocked — QE + Dev investigate same day
3
Gate 3 — Nightly Pipeline (11pm Mon–Fri) Full @regression E2E suite | Accessibility scan (axe-core) | Visual regression (Percy)
FAIL: Defect raised, owner assigned, fix by next business day
4
Gate 4 — Release Gate (release candidate) Performance vs NFR targets | Full DAST scan + dependency audit | Release Readiness Review
FAIL: Release blocked — RCA required before re-run
5
Gate 5 — Post-Deploy (production) @smoke suite against prod | Synthetic monitoring every 5 min | Canary: 5% traffic for 30 min
FAIL: Rollback triggered immediately by on-call engineer

Gate Ownership

GatePrimary OwnerEscalation
Gate 1 — PRDeveloperQE reviews failed coverage
Gate 2 — Post-mergeQE + DevTech Lead
Gate 3 — NightlyQEQE Lead
Gate 4 — ReleaseQE Lead + Release ManagerHead of Engineering
Gate 5 — ProductionDevOps + QEOn-call engineer

12Release Readiness Review

The Release Readiness Review (RRR) is a structured QE gate run before every production deployment. It produces a GO / NO-GO / CONDITIONAL GO verdict with documented evidence.

Release Entry Criteria (must confirm before review begins)

  • All stories in the release are in Done status in Jira
  • All stories have passed QA sign-off (DoD complete)
  • Release candidate build deployed to Staging environment
  • Release notes drafted and reviewed by Product Owner
  • Rollback plan documented and confirmed with DevOps

Defect Gate

SeverityOpen Count AllowedAction if Exceeded
🔴 Critical0Hard NO-GO — fix and re-run the entire review
🟠 High0NO-GO unless PO formally accepts risk in writing
🟡 MediumUp to 3 with PO sign-offDocument in release notes; fix in next sprint
🟢 LowAny numberDocument in release notes

Non-Functional Release Checks

⚡ Performance

  • Load test executed against release candidate
  • p95 response time within NFR targets
  • No memory growth over 30-minute soak
  • Core Web Vitals within acceptable range

🔒 Security

  • DAST scan against Staging — zero new High/Critical findings
  • Dependency audit — zero known critical CVEs
  • All @security scenarios passed
  • Auth/session tests passed for changed flows

♿ Accessibility

  • Automated axe scan on all new/changed pages
  • Zero critical violations reported
  • Keyboard navigation verified for new UI
  • Screen reader spot-check on high-traffic pages

🌐 Environment

  • Staging confirmed at release candidate version
  • Feature flags set correctly for production
  • DB migrations tested and reversible
  • Rollback plan documented and verified
  • Monitoring and alerting active for production

Post-Release Verification (within 2 hours of deployment)

  • @smoke suite re-run against production — all critical journeys pass
  • Synthetic monitoring green for all critical journeys
  • Error rate in production monitoring within normal baseline
  • Canary: 5–10% traffic deployed first — monitor 30 min before full rollout
  • Support team briefed on new features and known issues

15Rovo Agent Setup Using SKILL.md

Setup Steps

Upload SKILL.md to ConfluenceCreate a "QE Framework" Confluence space. Paste each major section of SKILL.md as a separate child page — this lets subagents focus on specific sections.

Create the Master QE Framework AgentJira → Chat → Agents → Create. Name: "QE Framework Agent". Add core instructions, knowledge sources (Confluence space + Jira project), and 5 tools.
3

5 Verification Test Prompts

TestWhat to TypeExpected Response
1"Review this story: As a customer I want to reset my password. AC: 1. User receives reset email."NOT READY — missing negative path, test data, Three Amigos. Blockers with owners listed.
2"Generate Gherkin for: filter products by category. AC: 1. Filter shows matching products. 2. No filter shows all."Feature file with @happy-path, @negative, @edge-case, Scenario Outline with data table.
3"Release readiness: v2.5.0. Regression 100%. 0 Critical. 0 High. 2 Medium PO accepted. p95 1.8s vs 2s target."GO verdict with structured report and QE Lead / Release Manager sign-off line.
4"Is this defect ready for a dev? Title: Login broken. Description: Login doesn't work on mobile."NOT READY — missing steps, expected vs actual, environment, test data, linked story.
5"What security scenarios should I write for a new admin user management API?"OWASP-aligned scenarios: auth tests, role access, injection, logging, session management.

16Repository Folder Structure

All QE test code follows this structure so any team member can navigate it without guidance. Test code is production code — the same PR review standards apply.

│   ├── features/                   ← BDD Gherkin feature files (by domain, not by story)
│   ├── step-definitions/           ← Automation step definitions (mirrors feature domains)
│   ├── unit/                       ← Unit tests (co-owned with Dev)
│   ├── functional/                 ← Functional / component test suites

│   ├── nightly.yml                 ← Gate 3: full E2E regression + a11y + visual

Key Rules

  • Test code is production code — same PR review standards apply
  • Nothing hardcoded — all URLs, credentials, and data via config
  • Gitignore reports/ — results stored in CI artefacts, not the repo
  • One feature file per domain area — not one per Jira story
  • Shared steps in support/ — never duplicate step definitions
  • Seeds are idempotent — safe to run twice, no duplicates
  • Factories over fixtures — programmatic data beats static JSON
  • Templates in .jira/templates/ — Rovo agent always loads these

Release Test Planning — QE Lead + Release Manager. Identify all stories in release. Define regression scope. Confirm environment availability. Schedule perf/sec/a11y tests.

Release Day
